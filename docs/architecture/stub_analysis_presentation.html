Resumo Executivo

!!
Padrao sistematico: A evolucao V1 → V2 trocou integracoes reais (Tasy API, ANS API, FHIR, TISS) por chamadas DMN + stubs como fallback. O resultado e que workers "funcionam" (retornam TaskResult.success) mas nao fazem integracao nenhuma com sistemas externos.
Aspecto V1 Workers V2 Workers
API clients Exigidos no construtor Optional, default=None
Chamadas externas Diretas: await self._ans.validate() Nenhuma ou via service com fallback stub
Logica de negocio Inline (100-400 linhas Python) Delegada a DMN table (regras estaticas)
Error handling Retry com backoff, classificacao de erro DMN routing: PROSSEGUIR / REVISAR / BLOQUEAR
Se client=None Crash (TypeError: NoneType not callable) Silenciosamente usa stub
Resultado visivel Dados reais do Tasy/ANS/FHIR "Stub submission accepted", "stub_template_42"
i
4 stubs ativos como default em producao:
(1) StubWhatsAppClient em 6 workers de billing/collection — notificacoes nunca enviadas
(2) StubMvSoulClient em sync_erp_data_worker — sincronizacao fake
(3) StubTUSSSuggestionEngine — sugestao de TUSS com 17 keywords vs milhares de codigos
(4) StubCID10SuggestionEngine — sugestao de CID-10 com 17 keywords vs ~70.000 codigos
i
Caminho de correcao sugerido:
(1) DI Container / Factory: Injetar clientes reais no startup do container, nunca permitir fallback silencioso para stub
(2) Health check de integracao: /health deve reportar se esta usando stub ou cliente real
(3) Recuperar logica V1: Re-integrar chamadas API reais nos services extraidos (nao nos workers)
(4) Feature flags: Controlar integracao real vs stub explicitamente (nao por client or Stub())

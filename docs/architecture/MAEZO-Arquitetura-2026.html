01 — Visão Geral

O que é o MAEZO?

O MAEZO é a plataforma de orquestração de ecossistemas de saúde da rede Austa. Atua como camada inteligente entre os sistemas legados dos hospitais (Tasy ERP, MV Soul), as operadoras de planos de saúde e os reguladores (ANS, ANVISA), automatizando processos clínicos e financeiros que hoje dependem de trabalho manual.

🏥

👤

04 — Workers Python

Como os Workers Funcionam?

Os workers são imagens Docker pré-construídas que rodam autonomamente — fazem poll no CIB Seven, executam a lógica delegando às tabelas DMN e devolvem o resultado. ~80% das mudanças de regra não exigem código Python: o time de negócio edita tabelas DMN no CIB Seven sem precisar de deploy.

Os 3 Caminhos de Uso

🎯

Arquétipos Genéricos (zero código)

7 workers pré-construídos cobrem 80%+ dos casos. Configura-se apenas via topic_registry.yaml + tabela DMN. Nenhuma linha Python.

🔧

Workers Customizados (25–90 linhas)

Para lógica verdadeiramente única (ex.: gerar XML TISS, criar resource FHIR). ~60% é boilerplate herdado da classe base. O desenvolvedor escreve apenas a lógica de domínio.

📋

Alteração de Regra (só DMN)

Mudar limiar de alerta, valor de glosa, protocolo clínico? Edita-se a tabela DMN no CIB Seven. Zero Python. Zero deploy. Efeito imediato.

Anatomia de um Worker Customizado

Python — worker customizado típico (~50 linhas)
@worker(topic="rc.billing.tiss_xml_generation")
class TissXmlGenerationWorker(BaseExternalTaskWorker):
    """Gera XML TISS 4.0 para submissão à operadora."""

    async def execute(self, context: TaskContext) -> TaskResult:
        # 1. Receber variáveis do processo (vindas do BPMN)
        conta_id    = context.variables.get("contaId")
        operadora   = context.variables.get("operadoraCodigo")
        tenant_id   = context.tenant_id

        # 2. Consultar DMN para regras de formatação da operadora
        #    (time de negócio edita essa tabela sem tocar Python)
        fmt_rules = await self.evaluate_dmn(
            context,
            "tiss-format-rules",          # tabela DMN no CIB Seven
            { "operadora": operadora }
        )

        # 3. Lógica única — geração do XML (não cabe em DMN tabular)
        conta  = await self.fhir_client.get_claim(conta_id, tenant_id)
        xml    = build_tiss_xml(conta, fmt_rules["versao"], fmt_rules["layout"])

        # 4. Retornar resultado para o processo BPMN
        return TaskResult.success({
            "tissXml":      xml,
            "status":       "GERADO",
            "operadoraCod": operadora,
        })

⚡
Stateless · Auto-descobertos · Horizontalmente escaláveis

Para adicionar um novo worker: criar o arquivo com @worker(topic="..."). Para escalar: aumentar replicas no Swarm. O CIB Seven distribui tarefas via lock sem coordenação manual.

Por que remover Basic Auth?

🔑
O código já usava Basic Auth internamente

O worker_runner.py (linhas 402–409) já configurava auth_basic com as credenciais do Basic Auth. As variáveis tinham nome "KEYCLOAK" mas o protocolo já era Basic Auth. Renomear para CIB7_USER / CIB7_PASSWORD elimina o Basic Auth sem alterar comportamento. O CIB Seven suporta Basic Auth nativamente via Spring Security.

Por que remover Redis?

🗄️
Redis tinha apenas 1 uso: deduplicação de webhooks

Todo o uso era SET nx ex com chave webhook:{system}:{key} e TTL de 7 dias. Substituído por INSERT ... ON CONFLICT DO NOTHING no PostgreSQL RDS com pg_cron para limpeza automática. Mesma semântica, zero infraestrutura adicional.

Por que n8n em vez de manter o código Python?

🔄
CDC Bridge tinha apenas 82 linhas de lógica de negócio

Era um roteador simples: tópico Kafka → lookup em tabela → POST HTTP para CIB Seven. O n8n substitui isso com 2 nodes (Kafka Trigger + HTTP Request), sem código, com interface visual, retry automático e dead-letter via PostgreSQL. A equipe de negócio pode ajustar mapeamentos sem precisar de um desenvolvedor Python.

Por que Docker Swarm em vez de Kubernetes?

🐳
Complexidade operacional reduzida drasticamente

O Kubernetes exigia EKS + Helm + Kustomize + cert-manager + nginx Ingress + 16 templates YAML. Docker Swarm usa o mesmo docker-compose.yml com campo deploy:, secrets nativos, rede overlay automática e Traefik para SSL. Para 14 serviços numa única VM, Swarm é a escolha operacionalmente mais saudável.

Deploy em 3 comandos

bash
# 1. Inicializar Swarm
docker swarm init --advertise-addr $(curl -s ifconfig.me)

# 2. Gerar CLUSTER_ID válido para Kafka KRaft (UUID base64 — obrigatório)
KAFKA_CLUSTER_ID=$(docker run --rm confluentinc/cp-kafka:7.7.0 \
  kafka-storage random-uuid)
# resultado: ex. MkU3OEdJTlQwSU9BMUVBQg  (22 chars base64)

# 3. Criar secrets (senhas nunca em variáveis de ambiente)
echo "senha_rds_segura_aqui"  | docker secret create postgres_password -
echo "senha_cib7_segura_aqui" | docker secret create cib7_admin_password -
echo "chave_n8n_32chars_aqui" | docker secret create n8n_encryption_key -

# 4. Deploy completo
RDS_HOST=maezo.xxx.rds.amazonaws.com \
RDS_USER=maestro \
KAFKA_CLUSTER_ID=${KAFKA_CLUSTER_ID} \
DOMAIN=austa.com.br \
IMAGE_TAG=latest \
docker stack deploy -c docker-compose.swarm.yml maestro

# 5. Registrar conector Debezium (após serviços subirem ~60s)
curl -X POST http://localhost:8083/connectors \
  -H "Content-Type: application/json" \
  -d @config/debezium/oracle-connector.json

# Verificar
docker service ls
⚠️
Sobre o Debezium e o Tasy Oracle

O Debezium Connect precisa de acesso ao banco Oracle do Tasy ERP (host, porta, credenciais, LogMiner habilitado). Isso requer configuração prévia com o time de DBA responsável pelo Tasy. O connector Debezium é registrado via API REST após o serviço estar rodando: POST http://debezium:8083/connectors com o payload de configuração do Oracle connector.

Problema 1 — DMNs com Inputs que Ninguem Fornece

42% dos DMN files (518 de 1.231) recebem booleans pre-calculados sem componente produtor
!!
O que acontece em runtime: O DMN engine avalia a regra com o input = null (nao fornecido). Dependendo do hitPolicy, isso pode: (a) cair na regra fallback "Revisar", gerando falso positivo de revisao manual; (b) nao casar com nenhuma regra e retornar vazio; (c) casar com regra errada se null for tratado como false.
1

revenue_cycle/dmn/pricing/validation/procedure_code_adjudication.dmn

O DMN de adjudicacao de codigo de procedimento espera 2 booleans de validacao contra sistemas externos (TASY e ANS), mas nenhum worker, service, ou adapter faz essas chamadas.

3
contratoPermiteSeparar / mesmoAtoAnestesico — Contrato + Prontuario

DMN de validacao de cobranca agrupada (mastectomia radical) exige 2 booleans que dependem de: (a) consultar clausula especifica no contrato do convenio; (b) cruzar registros de ato anestesico no prontuario cirurgico.

Impacto: Cobrancas agrupadas/separadas nunca sao validadas automaticamente. Hospital pode cobrar separado quando deveria ser agrupado (glosa) ou vice-versa (perda de receita).
4
codigoAnvisaValido / loteRegistradoProntuario — OPME Rastreabilidade
revenue_cycle/dmn/billing/opme/bill_opme_001.dmn

Validacao de OPME implantavel exige 4 inputs pre-calculados: validacao ANVISA, registro em prontuario, dias ate validade, e completude do registro.

4 INPUTS PRE-CALCULADOS
codigoAnvisaValido (boolean) <-- API ANVISA ou tabela referencia loteRegistradoProntuario (boolean) <-- Cross-ref prontuario diasAteValidade (number) <-- Calculo: expiry_date - today registroPacienteCompleto (boolean) <-- Checagem de completude

Resumo Executivo

Falhas Criticas (Runtime)

  • 22/22 decisionRef quebradas — Toda businessRuleTask falha com DMN_DECISION_NOT_FOUND
  • 2 dominios DMN inexistentes — clinical_documentation_completeness e revenue_optimization_strategy
  • ~55 topics sem worker — Instancias ficam presas indefinidamente
  • 4 topics com prefix mismatch — denial.* vs glosa.* (worker existe mas nao conecta)

Falhas Estruturais (Design)

  • 743 inputs boolean orfaos — DMNs esperam dados pre-calculados sem componente produtor
  • 42% dos DMNs afetados — 518 de 1.231 files com boolean sem fonte
  • Camada de enriquecimento ausente — Nao existe servico que converte dados brutos em inputs DMN
  • Naming inconsistente em 3 camadas — BPMN, DMN e Worker usam convencoes diferentes
i
Causa raiz: Os 3 artefatos (BPMN, DMN, Worker) foram desenvolvidos de forma independente, sem contrato de interface compartilhado. O BPMN usa nomes conceituais (billing_bundle_rules), os DMNs usam IDs tecnicos numerados (Decision_BILL-BUNDLE-001), e os Workers usam caminhos do filesystem ("bundle/bill_bundle_001"). Nao existe uma camada de resolucao/alias que unifique os tres.
i
Caminho de correcao sugerido: (1) Definir convenção unica de naming para decision IDs — o decisionRef do BPMN deve ser o decision id do DMN. (2) Criar camada de "Data Enrichment Worker" que pre-calcula booleans antes da avaliacao DMN. (3) Alinhar topic naming entre BPMN e Workers (ex: todos com prefixo subdominio.acao). (4) Criar DMNs ausentes (clinical_documentation_completeness, revenue_optimization_strategy).
